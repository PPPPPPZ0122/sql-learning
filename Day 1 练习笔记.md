# 🎯 Day 1——25 题：基础语法输入、空值陷阱与数值修约的底层硬核重构

## 1. 现场还原

在今天 Day 1 的 25 题极限马拉松中，高频遇到了限制明细、空值清洗、条件分支以及数值精度控制的真实业务场景。以下是今天肉身蹚坑的典型初始代码与逻辑记录：

### 🚨 场景一：运营限制明细行数误用 `WHERE`

- **初始逻辑**：现在运营只需要查看前 2 个用户明细设备 ID 数据。
```
-- 初始做法：使用 WHERE 对自增 id 进行硬编码限制
SELECT device_id
FROM user_profile
WHERE id <= 2;
```

### 🚨 场景二：年龄空值清洗出现“野路子”代码

- **初始逻辑**：剔除没有获取到年龄的用户，取出所有年龄值不为空的数据。
```
-- 初始做法：使用 age > 0 或者是 age != NULL 进行空值过滤
SELECT device_id, gender, age, university
FROM user_profile
WHERE age > 0;         
```

### 🚨 场景三：聚合结果筛选错写进 `WHERE`

- **初始逻辑**：计算各学校平均答题数，并筛选出平均答题数大于 5 的学校。

```
-- 致命报错做法：直接在 WHERE 后面拦截聚合函数结果
SELECT university, AVG(question_cnt) AS avg_cnt
FROM user_profile
WHERE AVG(question_cnt) > 5  -- 💥 触发报错：Invalid use of group function
GROUP BY university;
```

## 2. 致命原因与底层逻辑

大厂数据分析面试中，面试官最看重候选人是否具备**数据库底层执行器（Optimizer）的角度**来审视代码。今天在 25 题中踩到的每一个细碎坑，背后都有着严密的底层逻辑：

### 🧠 逻辑一：`LIMIT` 与 `WHERE` 的本质区别

`WHERE` 是行级过滤器，它在数据从磁盘加载到内存的过程中，逐行扫描并根据条件判定 True/False。而运营的需求是“只要前两行明细”，属于标准的**结果集切片**。用 `id <= 2` 属于硬编码，一旦未来底层物理主键 `id` 发生断层或重新排序，业务数据将彻底失真。限制结果集行数，必须使用专门的语法 `LIMIT`。

### 🧠 逻辑二：MySQL 里的三值逻辑（Three-Valued Logic）与 NULL 陷阱

为什么不能写 `age != NULL`，而用 `age > 0` 极其危险？

- **底层原理**：在 SQL 中，`NULL` 代表“未知”（Unknown）。任何常规运算符（`=`, `!=`, `<`, `>`）与 `NULL` 进行比较，其返回结果全部都是 `Unknown`。因此，`WHERE age != NULL` 底层恒为假，一条数据也查不出来。
    
- **业务漏洞**：用 `age > 0` 过滤空值属于偏门野路子。如果业务数据中存在异常脏数据（如 `age = -1` 或者是未填写的默认字符串 `''`），`age > 0` 会直接改变原有的业务清洗口径，漏掉数据。必须严格使用 `IS NOT NULL` 或判断空字符串 `!= ''`。
    

### 🧠 逻辑三：SQL 核心算子执行顺序决定了 `WHERE` 无法拦截聚合函数

为什么聚合函数结果作为筛选条件时，不能用 `WHERE` 而是用 `HAVING`？因为数据库在执行一条 SQL 时，底层的真正生命周期如下：

1. `FROM`（加载目标表数据）
    
2. `JOIN`（关联多表）
    
3. `WHERE`（**行级过滤，此时数据还没分组，聚合函数如 `AVG()` 根本还没开始计算！**）
    
4. `GROUP BY`（**开始进行数据分组，数据开始分桶**）
    
5. `HAVING`（**组级过滤，聚合函数在此时才大摇大摆地在各自的桶里进行计算和筛选**）
    
6. `SELECT`（提取列、重命名/别名赋值）
    
7. `ORDER BY`（最终排序，**注意它处于 GROUP BY 之后，只是对最终输出结果集进行排序，切勿与分组混淆**）
    
8. `LIMIT`（切片输出）
    

> [!important] 关键结论
> 
> 因为 `WHERE` 的执行顺序死死压在 `GROUP BY` 之前，所以当 `WHERE` 执行时，聚合函数还没出生呢！这就是为什么必须将聚合过滤条件移到 `HAVING` 后面。并且在 MySQL 环境下，`HAVING` 完美支持配合 `SELECT` 阶段定义的列别名（重命名）直接进行过滤。

### 🧠 逻辑四：运算符优先级与模糊匹配规范

- **AND 优先级强于 OR**：在处理多条件组合查询时，`AND` 的执行优先级高于 `OR`。题解虽然说“可以不用套括号”，但是在复杂业务中，为了**代码的可读性以及防止后续维护时逻辑滑坡**，大厂生产标准强制要求将 `OR` 的两端套上明确的括号 `()`。
    
- **等号 `=` 与 `LIKE` 的降维打击**：等号 `=` 是**精确的全字等值匹配**。如果你的条件中带有通配符 `'%_%'`（百分号代表任意多字符，下划线代表单个任意字符），等号会把 `'%_%'` 当作一个普通的纯文本字符串去死磕匹配，从而导致查空。必须使用 `LIKE` 算子，才能激活数据库的**模式匹配（Pattern Matching）引擎**。

## 🛠️ 正确解决方案（大厂生产标准）

基于上述底层逻辑，重构今天的所有踩坑代码，对齐大厂标准规范：

### ✅ 规范解法一：明细切片使用标准 `LIMIT
```
SELECT device_id
FROM user_profile
-- 💡 生产标准：抛弃硬编码，直接用 LIMIT 限制返回的结果行数
LIMIT 2;
```

### ✅ 规范解法二：使用标准 `IS NOT NULL` 剔除空值
```
SELECT device_id, gender, age, university
FROM user_profile
-- 💡 生产标准：严格使用符合三值逻辑的专门算子，防范空字符串和负数异常
WHERE age IS NOT NULL 
  AND age != '';
```

### ✅ 规范解法三：聚合结果走 `HAVING` 并配合列别名
```
SELECT 
    university, 
    AVG(question_cnt) AS avg_cnt
FROM user_profile
GROUP BY university
-- 💡 生产标准：在 GROUP BY 之后使用 HAVING 过滤聚合结果，直接引用别名 avg_cnt
HAVING avg_cnt > 5;
```

### ✅ 规范解法四：模糊匹配、取反（NOT LIKE）与运算符优先级规范
```
SELECT device_id, university
FROM user_profile
-- 💡 生产标准：1. 使用 LIKE 激活通配符匹配；2. 多条件组合时用括号显式隔离 OR 逻辑
WHERE (university LIKE '%浙江%' OR university LIKE '%北京%')
  AND university NOT LIKE '%学院%';
```

### 🔢 附：数值修约军规 `ROUND(X, D)` 函数

在做数据看板（BI）或者汇报指标时，经常需要对数值进行保留小数操作：

```
-- 1. decimals > 0：正常四舍五入到指定小数位
SELECT ROUND(125.315, 2); -- 结果：125.32

-- 2. decimals = 0：四舍五入到最接近的整数（抹去小数部分）
SELECT ROUND(125.515, 0); -- 结果：126

-- 3. decimals < 0：在小数点左侧（整数位）进行四舍五入修约
SELECT ROUND(125.315, -1); -- 结果：130（个位数 5 往十位进一）
SELECT ROUND(123.315, -1); -- 结果：120（个位数 3 被四舍五入抹零）
```

## 💡 转行兵法：风控策略运营与数据分析的面试无敌防线

> [!danger] 面试官视角：用“执行顺序”和“数据失真”反杀八股文
> 
> 当面试官在面试中抛出：**“请你说说 WHERE 和 HAVING 的区别”**。
> 
> **❌ 60分普通回答**：
> 
> “WHERE 是在分组前过滤，后面不能接聚合函数；HAVING 是在分组后过滤，后面可以接聚合函数。”
> 
> **🔥 100分高级数析专家话术（结合内容审核背景）**：
> 
> “从底层执行引擎的角度来看，两者的根本差异在于**SQL算子的执行流顺序**。`WHERE` 作用于 `GROUP BY` 之前，是行级扫描，此时聚合函数的内存桶尚未建立；而 `HAVING` 作用于 `GROUP BY` 之后，专门用于拦截组级特征。
> 
> 在我之前的内容审核业务场景中，这种执行顺序直接决定了数据指标的生死。比如我们要筛选**‘昨日人均审核量大于500条的非白名单审核小组’**：
> 
> 1. 我们必须首先使用 `WHERE` 在最前端把‘VIP白名单企业’或者‘机审自动过审’的明细行过滤掉（降低后续算子的内存压力）；
>     
> 2. 接着通过 `GROUP BY team_id` 按照审核小组切分数据块；
>     
> 3. 最后在 `HAVING COUNT(doc_id) / COUNT(DISTINCT auditor_id) > 500` 阶段，利用已经计算出来的组级聚合指标，精准筛选出人效爆仓的团队。
>     
> 
> 如果分不清两者的执行流，错把组级过滤写进 `WHERE`，不仅会遭遇数据库抛出 `Invalid use of group function` 的编译中断，更会在错综复杂的风控指标计算中，导致严重的**数据大面积失真与逻辑穿透**。”
